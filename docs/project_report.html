Quantphemes RL Trading Bot: Architecture and Readiness Report

This report summarizes the implemented codebase, current synthetic-data performance, live API readiness, and the Bloomberg data needed before using the model for real trading decisions.

1. Implementation Status

All planned phases 0-8 are implemented: skeleton, data adapters, portfolio/env, plugins, experiment runner, diagnostics/reporting, experiment YAML grid, live bot, and federated Q-table merge.

2. Architecture

experiments/*.yaml -> config.py -> plugin registry
                           |
                           +-> data_source -> DayData
                           +-> state_encoder -> MarketContext -> state
                           +-> reward_function -> reward
                           +-> agent -> action/update/save/load

apps/run_experiment.py -> results/<run>/metrics.json, q_state.pkl, report.html
apps/bot.py            -> artifacts/q_state.pkl + Quantphemes API -> holding PATCH
apps/federate.py       -> q_state.pkl + partner q_state.pkl -> merged q_state.pkl

5. Bloomberg Data Required

Export one file per asset/interval into data/raw/. Preferred filenames match the experiment YAMLs, e.g. 2800_HK_1h.csv, 7299_HK_30min.csv.

FieldRequirement
Date/TimeTimezone-aware or Hong Kong local timestamps. Need bars at every configured decision time and close time.
Open, High, Low, Close/Last_PriceOHLC price fields. Bloomberg XLSX adapter accepts sheet Data with Date, Open, High, Low, Last_Price, Volume.
VolumePreferred; optional for the current baseline but useful for future state encoders.
AssetsAt minimum 2800.HK. Tier 2 needs 2828.HK, 7299.HK, 7568.HK if available.
Intervals1-hour, 30-minute, 5-minute, and EOD. If Bloomberg only gives 1-minute bars, we can aggregate later.
History lengthPrefer 3-5 years. Minimum useful lab run: 1 year of complete trading days.
