05
Leveraged ETF Rationale

Why more paper slots use 7226.HK

Fee hurdle

The round trip cost is about 40 bps. Many 2800.HK intraday moves are smaller than that, so cash can be a rational learned action.

Research signal

Leveraged ETFs can move more intraday, giving the Q-table more room to learn whether exposure is worth the fee.

2800.HK remains the broad HSI benchmark. Leveraged ETFs are used more heavily for paper observation because their larger movement makes strategy behavior easier to distinguish.

07
Architecture

Shared core, thin applications

Config

YAML plus Pydantic validation define the whole experiment or production bot.

Plugins

Data source, encoder, reward, and agent are selected through a registry, not hard-coded branches.

Core

TradingEnv, Portfolio, StateEncoder, RewardFunction, and Agent are shared by training and live execution.

Flow: YAML config → registry → data/state/reward/agent → training report + Q-table → live bot → Quantphemes API.

08
Data and Configuration

Strategy behavior is declared, validated, and reproducible

Data sources

The current research data mainly comes from Webull and Futu exports/downloads, normalized into HK-time bars.

YAML configs

Asset, date range, fee, decision times, encoder, reward, agent, and artifacts are visible in one file.

Pydantic validation

Bad config or malformed data fails early instead of silently entering training or live execution.

Supported adapters

Bloomberg XLSX is supported for future exports, but it was not the main source for the current paper fleet.

11
Training Loop

How a Q-table is produced

Walk-forward training

Historical days are split into training and test windows. Each training episode shuffles days and starts each day with a fresh portfolio.

Decision steps

At each decision time, the environment measures NAV before trade, executes action, applies fees, advances time, and computes reward.

Q update

The agent updates one cell: decision point, state, action. Visit counts determine the per-cell learning rate.

Artifact output

The selected result folder contains q_state.pkl, config snapshot, metrics, logs, and HTML diagnostics.

12
Live Runtime

How one paper decision is made

  1. Fetch current price and broker cash/quantity.
  2. Build market context: current price, open, previous close, previous decision price.
  3. Encode context into a discrete state.
  4. Read the Q-table row for this decision point and choose the greedy action.
  5. Convert action into target quantity using lot size and available cash.
  6. Patch Quantphemes holding and write one structured JSON log line.

Trading Rules

Financial assumptions are explicit and tested

20 bps

One-side transaction fee, applied only when position changes.

EOD

Live bot targets flat quantity near the close; each training day starts fresh.

Lot size

Target quantity respects exchange lot constraints and available cash.

Diagnostics

We diagnose the model, not only the final NAV

Coverage

Which decision/state/action cells were actually visited during training.

Oracle

Perfect-foresight upper bound under the same fee model.

Signal Checks

Feature bins, conditional returns, and mutual information.

17
Live Operations

Production-style deployment is observable and reversible

Server

Azure VM, Python virtual environment, one systemd service per bot, restart policy enabled.

Logs

One JSON line per decision: price, state tuple, action, target quantity, current quantity, and status.

Secrets

API key and portfolio/strategy IDs are in server-side .env, never committed.

Rollback

Stop a service, restore a previous Q-table, restart only that bot.

18
Quantphemes API Integration

Broker behavior shaped the deployment design

  • Documented endpoints are wrapped in api_client/quantphemes.py.
  • Undocumented price endpoints are isolated in api_client/_undocumented.py.
  • Holding PATCH calls are tested through a sync helper with mocked network tests.
  • Symbol tradeability is validated separately from price availability.

21
Current Boundaries

What the report does and does not claim

It claims

The research-to-paper pipeline is implemented, tested, documented, and able to run multiple Q-table strategies.

It does not claim

The current paper strategies are production-profitable. Live observation and broker behavior still matter.

Backtests include fees, but they should be read as candidate filters and diagnostics, not final investment evidence.
