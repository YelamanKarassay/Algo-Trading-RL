Quantphemes RL

Q-Learning Paper-Trading Fleet

A reproducible research, diagnostics, and live deployment system for Hong Kong ETF trading strategies.

Speaker notes

Open by framing this as an engineering system, not just a single trading script. The goal is to show that research, backtesting, diagnostics, and live paper execution are connected and auditable.

01
Why This Project Exists

One workflow from hypothesis to paper trading

Research

Run repeatable historical experiments across ETF assets, intervals, features, rewards, and agents.

Explain

Use reports, oracle checks, coverage, and fee diagnostics to understand why the model behaves a certain way.

Deploy

Promote selected q_state.pkl artifacts to paper trading through the Quantphemes API.

Speaker notes

The important message is continuity. We do not train one way and trade another way. The same state encoders and agent artifacts are used in both tracks.

05
Architecture

Shared core, thin applications

Config

YAML plus Pydantic validation define the whole experiment or production bot.

Plugins

Data source, encoder, reward, and agent are selected through a registry, not hard-coded branches.

Core

TradingEnv, Portfolio, StateEncoder, RewardFunction, and Agent are shared by training and live execution.

Flow: YAML config → registry → data/state/reward/agent → training report + Q-table → live bot → Quantphemes API.

Speaker notes

This slide explains maintainability. A new idea usually means a new YAML or plugin, not editing the live bot.

06
Framework Choices

Why these tools?

Python, NumPy, Pandas

Best fit for market data processing, vectorized metrics, and quick research iteration.

Pydantic + YAML

Readable experiment files with strict validation and inheritance.

Pytest + Ruff

Fast regression testing and style checks before deployment.

MkDocs Material

Searchable, polished documentation for mixed technical and non-technical readers.

Speaker notes

Connect each framework to project risk reduction: validation prevents config mistakes, tests protect trading rules, docs make operations repeatable.

07
Trading Rules

Financial assumptions are explicit and tested

20 bps

One-side transaction fee, applied only when position changes.

EOD

Live bot targets flat quantity near the close; each training day starts fresh.

Lot size

Target quantity respects exchange lot constraints and available cash.

Speaker notes

These rules matter because small intraday ETF moves can be smaller than round-trip fees. The environment was built to make that visible.

08
Diagnostics

We diagnose the model, not only the final NAV

Coverage

Which decision/state/action cells were actually visited during training.

Oracle

Perfect-foresight upper bound under the same fee model.

Signal Checks

Feature bins, conditional returns, and mutual information.

Speaker notes

This is where we separate feature weakness from policy weakness. Without diagnostics, a bad result is hard to interpret.

09
Live Operations

Production-style deployment is observable and reversible

Server

Azure VM, Python virtual environment, one systemd service per bot, restart policy enabled.

Logs

One JSON line per decision: price, state tuple, action, target quantity, current quantity, and status.

Secrets

API key and portfolio/strategy IDs are in server-side .env, never committed.

Rollback

Stop a service, restore a previous Q-table, restart only that bot.

Speaker notes

Mention that this is designed for safe paper-trading operations. It is not a notebook that must stay open on a laptop.

10
Quantphemes API Integration

Broker behavior shaped the deployment design

  • Documented endpoints are wrapped in api_client/quantphemes.py.
  • Undocumented price endpoints are isolated in api_client/_undocumented.py.
  • Holding PATCH calls are tested through a sync helper with mocked network tests.
  • Symbol tradeability is validated separately from price availability.
Speaker notes

This is a good place to ask Quantphemes about symbol whitelists, paper execution visibility, and any limitations on strategy automation for the current plan.
